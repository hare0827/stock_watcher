데이터 구조 접근 방식

보유 내역을 어떻게 저장하고 관리할지 선택해주세요

A

독립적인 holdingsStore 추가 추천

기존 stocksStore와 별개로 새 Zustand 스토어 holdingsStore를 만들어요. Map<ticker, Holding[]> 구조로 종목별 매수 내역 배열을 관리하고 AsyncStorage에 @holdings 키로 저장해요.

장점

  • 기존 코드 변경 없음
  • 역할 분리 명확
  • alertStore 패턴과 동일

단점

  • 파일 하나 더 생김
B

Stock 타입에 holdings 필드 추가

기존 Stock 인터페이스에 holdings?: Holding[] 필드를 추가하고 stocksStore 안에서 함께 관리해요.

장점

  • 스토어 하나로 통합
  • 종목 삭제 시 자동 정리

단점

  • Stock 타입이 무거워짐
  • 기존 stocksStore 대규모 수정
  • AsyncStorage 포맷 변경 (마이그레이션 필요)
C

상세 화면 로컬 상태 + AsyncStorage 직접 접근

글로벌 스토어 없이, 상세 화면 컴포넌트 안에서 useState + useEffect로 AsyncStorage를 직접 읽고 써요.

장점

  • 스토어 레이어 불필요

단점

  • 상태가 화면에 종속됨
  • 다른 화면에서 접근 불가
  • 기존 패턴과 불일치
